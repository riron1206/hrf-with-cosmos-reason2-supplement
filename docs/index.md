# Hierarchical Reasoning Framework for Dashcam Incident Analysis with Cosmos-Reason2-8B

A **Physical AI system** that watches dashcam video and produces structured incident reports — detecting *what* happened, *when*, *who* was involved, and *why* — using **NVIDIA Cosmos-Reason2-8B** as the sole inference model, with no fine-tuning.

## Problem &

Impact

Every day, millions of hours of dashcam footage are recorded worldwide, but extracting actionable insights from this video requires human reviewers who cannot scale. The challenge is hard because understanding traffic incidents demands three types of reasoning simultaneously:

* **Spatial reasoning** — who is where, road geometry, motion cues
* **Temporal reasoning** — when the critical moment begins
* **Causal reasoning** — why it happened, not just what happened

This project solves this by automating the full analysis pipeline. The output is a structured report that answers all key questions without watching the original video:

| Question | Output Field | Example |
|----------|-------------|---------|
| **When?** | Incident start frame | Frame 159 |
| **What?** | Incident type + severity | Ego-car collision, severity level 1 |
| **Who?** | Participant counts | 1 vehicle, 0 pedestrians, ego involved |
| **Why?** | Causal explanation | "A black car suddenly moved into the lane, leading to a collision" |

This pipeline has been validated on **661 real dashcam videos** from the [Kaggle 2COOOL challenge](https://www.kaggle.com/competitions/2coool), processing them end-to-end with 8×H100 GPUs. No specialized sensors or hardware are required beyond standard dashcam footage — making the system immediately applicable to the billions of dashcam-equipped vehicles already on roads worldwide.

**Real-world applications**:
- **Autonomous vehicles**: Automatically detect and annotate edge cases (collisions, near-misses) from recorded drives to improve perception and planning systems
- **Insurance**: Accelerate claim assessment with structured, machine-readable incident reports — replacing hours of manual video review per claim
- **Fleet safety**: Continuously monitor thousands of vehicles at scale — only incidents are surfaced with full context, enabling proactive safety interventions
- **Dashcam analytics platforms**: Provide structured, machine-readable incident data that integrates directly into existing dashcam cloud infrastructure
- **Traffic research**: Scale safety analysis far beyond what manual review allows

## Why

Cosmos-Reason2

[Cosmos-Reason2](https://github.com/nvidia-cosmos/cosmos-reason2) is NVIDIA's Physical AI reasoning model trained to understand physical common sense and generate appropriate reasoning for real-world scenarios. This project leverages three key capabilities:

1. **Extended Thinking**: The `<think>` reasoning format enables chain-of-thought analysis of complex traffic scenes, improving accuracy on multi-attribute classification (crash severity, involved participants, causal reasoning).
2. **Multi-Image Reasoning**: Cosmos-Reason2-8B can jointly reason over multiple consecutive frames, capturing temporal evolution of incidents rather than analyzing isolated snapshots.
3. **Physical World Understanding**: Pre-trained on physical-world data, the model excels at spatial reasoning (vehicle positions, pedestrian trajectories) and causal inference (why a collision occurred), which are critical for incident analysis.

**Design choice — single model, no fine-tuning**: Rather than assembling multiple specialized models or fine-tuning on task-specific data, this project uses Cosmos-Reason2-8B as the sole inference engine across all pipeline stages. This deliberate constraint serves two purposes: (1) it demonstrates the depth of Cosmos-Reason2's Physical AI reasoning — that a single model can handle captioning, temporal detection, counting, text generation, and logical reconciliation when guided by well-designed hierarchical prompts; and (2) it keeps the system simple to deploy, reproduce, and adapt to new domains without retraining.

## Solution:

Hierarchical Reasoning Pipeline

![](https://raw.githubusercontent.com/turingmotors/hrf-with-cosmos-reason2/main/assets/cosmos_fig2.png)

All inference steps use a single model: **Cosmos-Reason2-8B**. The key insight is to process video hierarchically — cheap text-only reasoning identifies *where* to look, then expensive multi-image reasoning analyzes *what* happened — avoiding the cost of sending all frames through the VLM. This hierarchical decomposition achieves two goals simultaneously: it reduces computational cost (only a few key frames go through multi-image reasoning) and improves accuracy (each stage focuses on one type of Physical AI reasoning — spatial perception, temporal detection, or causal explanation — rather than solving everything at once).

| Stage | Input | Output | Purpose |
|-------|-------|--------|---------|
| Frame Captioning | Individual frames | Per-frame text descriptions | Convert visual content into structured text |
| Frame Detection | Captions (text only) | Key frame index | Identify critical moments without re-processing images |
| Description | Multi-frame images around the key frame | Structured JSON (counts + text) | Produce detailed analysis with self-consistency voting |

## Technical

Implementation

#

## Stage

1: Frame Captioning

Each frame is a vertically stacked image: the **top half** is a driver gaze heatmap (white marks indicate focus), and the **bottom half** is the original dashcam frame.

The model systematically scans left → center → right, near → far, matches the gaze heatmap with the dashcam content, and outputs a structured JSON with 11 keys per frame:

| Key                                | Type   | Description                                              |
| ---------------------------------- | ------ | -------------------------------------------------------- |
| `Scene_Description`                | string | Road, vehicles, pedestrians, signs, lighting, occlusions |
| `Hazard_Clues`                     | string | Risk factors that could lead to accidents                |
| `Gaze_Focus`                       | string | What the driver's gaze overlaps with (from heatmap)      |
| `Incident_Detection`               | int    | 1 = accident, 0 = hazard, -1 = none                      |
| `Crash_Severity`                   | string | One of 7 predefined severity levels                      |
| `Ego_car_involved`                 | int    | 0 or 1                                                   |
| `Label`                            | string | One of 19 incident type classes                          |
| `Number_of_Bicyclists_Scooters`    | int    | Count of involved cyclists/scooters                      |
| `Number_of_Animals`                | int    | Count of involved animals                                |
| `Number_of_Pedestrians`            | int    | Count of involved pedestrians                            |
| `Number_of_Vehicles_excluding_ego` | int    | Count of other vehicles involved                         |

Frames are sampled every 10th frame (configurable via `frame_stride`) with low temperature (0.01) for deterministic per-frame analysis.

#

## Stage

2: Frame Detection with Self-Consistency

The per-frame CSV from Stage 1 is presented to the model. The model identifies which frame marks the start of the most severe incident.

**Self-Consistency Voting**: 12 independent samples are generated (split across multiple API calls of 4 each), and the incident frame is determined by majority vote. If the majority vote returns -1 (no incident), the median is used as a fallback.

**Adaptive Context Management**: If the per-frame CSV exceeds the model's context window (16,384 tokens), rows are automatically subsampled at increasing strides until the prompt fits.

#

## Stage

3: 3-Stage Description with Reconciliation

Using 4-8 consecutive frames around the detected incident frame, the model runs three specialized inference stages:

```
Stage 3a: Count    →  9 keys (numeric/categorical: severity, ego involvement, label, counts)
Stage 3b: Text     →  3 keys (caption before incident, reason of incident)
Stage 3c: Reconcile → 11 keys (merged output, corrected by image evidence)
```

**Why separate Count and Text?** Counting/classification and natural language generation are fundamentally different tasks. Separating them allows each stage to focus on its strengths, and the Reconcile stage can detect and fix contradictions between them.

**Reconciliation Rules**: The Reconcile stage enforces logical consistency:
- If `Incident Detection = -1`: severity must be "No Crash", all counts must be 0
- If `Incident Detection = 0` (hazard): severity must be "No Crash", label must be hazard-compatible
- If `Incident Detection = 1` (accident): severity and ego involvement must be consistent with the collision type
- Evidence precedence: Images > Text predictions > Count predictions

**Contradiction Detection** (`detect_contradictions()`): Automatically flags logical inconsistencies such as:
- Incident=1 but Crash Severity is "No Crash"
- Ego involved=1 but Crash Severity is "No Crash"
- Text mentions collision keywords but Incident Detection=-1

**Self-Consistency**: Each of the three stages independently generates multiple samples (n=4), aggregated via majority vote for categorical fields and median for numeric counts. Text fields are selected from the sample whose categorical predictions best match the aggregated result.

## Robustness

Features

The pipeline includes multiple layers of fallback to handle edge cases in model output:

| Feature                                  | Description                                                                          |
| ---------------------------------------- | ------------------------------------------------------------------------------------ |
| JSON rescue from reasoning               | Extract the last balanced `{...}` if JSON appears only inside `<think>`              |
| Retry with escalating repetition penalty | Retry up to 5 times if output is empty, increasing `repetition_penalty` (1.05 → 1.5) |
| Post-process null filling                | Fill remaining empty fields from intermediate outputs or safe defaults               |

## Scale &

Architecture

The pipeline has been validated on **661 dashcam videos** using distributed parallelism:

```
8 GPUs (H100 80GB each)
  └── 8 vLLM servers (1 per GPU, ports 8000-8007)
       └── 4 workers per server = 32 concurrent requests
            └── Round-robin scheduling across videos
```

Each vLLM server runs with:
- `--gpu-memory-utilization 0.85` for efficient memory use
- `--max-model-len 16384` context window
- `--reasoning-parser qwen3` for automatic `<think>` tag extraction
- continuous batching for high throughput

Shell scripts (`run_*_parallel.sh`) handle work distribution: each worker processes every N-th item (where N = total workers), ensuring balanced load without coordination overhead.

## Key

Features Summary

| Feature | Description |
|---------|-------------|
| **Single-Model Architecture** | Cosmos-Reason2-8B handles all inference stages — no auxiliary models, no fine-tuning |
| **Hierarchical Reasoning** | Frames → captions → key frame → multi-attribute description, each stage building on the previous |
| **Self-Consistency Voting** | n=12 (detection) / n=4 (description) independent samples aggregated via majority vote |
| **3-Stage Description + Reconciliation** | Count → Text → Reconcile with 20+ contradiction detection rules |
| **YAML-Configurable Prompts** | All prompts externalized for domain adaptation without code changes |
| **8-GPU Distributed Processing** | 32 concurrent workers with round-robin scheduling and continuous batching |
| **Blind A/B Testing Tool** | Randomized human evaluation web UI for comparing submissions |
| **Web Application** | Browser-based interface with real-time progress tracking and JSON export |

## Domain

Adaptation

All VLM prompts are externalized into YAML config files. To adapt the pipeline to a new domain:

1. Copy config: `cp default.yaml my_domain.yaml`
2. Edit prompts (incident taxonomy, severity rules, counting rules)
3. Run with `--config configs/my_domain.yaml` — no code changes needed
